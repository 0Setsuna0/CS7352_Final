## 0. Scope, Claims, and Honesty Contract（v3 扩到五条）

本项目实现 training-free 时空 token merging 并刻画 quality–efficiency Pareto 前沿。以下五条为军规：

1. **真加速。** Merge 必须缩短实际计算的序列长度。主交付物为 `scope="block"`（attention + FFN 全部在缩短序列上计算）。"merge 后立即 restore 再进 attention" 仅作为正确性调试路径。
2. **真时空。** 同时交付 spatial 与 spatiotemporal 两个变体，并量化差异。
3. **真硬件报告。** 必报 transformer-only 时间（CUDA events 测量）+ 端到端时间，并标注 offload 模式。
4. **真运动保持（v3 新增）。** 任何 "时间一致性更好" 的结论必须同时报告 `dynamic_degree`。**视频变静态会自动刷高所有 flicker 类指标，这是已知的指标作弊路径，必须用运动约束封死。** Pareto 前沿只在 `dynamic_degree ≥ baseline × 0.9` 的配置中绘制；不满足约束的配置单独列为 "motion-degraded" 并如实报告。
5. **真预算（v3 新增）。** 编码前先算 FLOPs 账（§2.1），明确加速上限；跑实验前先算 GPU-hours 账（§8），明确日历可行性。预期与实测的差距写进报告，不掩盖。

Out of scope 不变：不训练、不蒸馏、不动 VAE 和 text encoder。

## 2. Architecture Decision（v3 重写）

### 2.1 先算账：FLOPs 预算（编码前必读）

quality preset 下：latent 帧数 13（49 帧经 VAE 4x 时间压缩 + 首帧），patch 后空间网格 30×45，**视频 token N_v ≈ 17,550**，文本 token 226，总 N ≈ 17,776。CogVideoX-2B：d = 1920，30 层，FFN 4x。

每 token 每层 FLOPs（前向，2×MAC 计）：

| 组件 | 公式 | 数值 | 占比 |
|---|---|---|---|
| QKVO 投影 | 8d² | ≈ 29.5 M | ~13% |
| FFN | 16d² | ≈ 59.0 M | ~26% |
| Attention（QKᵀ + AV） | 4·N·d | ≈ 136.5 M | ~61% |

结论：在此序列长度下 attention 占 ~61%（N > 6d 的临界点之上）。由此推出各 scope 在 ratio=0.2 时的**理论 FLOPs 削减**：

- `attn_only`：attention 部分按 (0.8)² 缩放 → 削减 ≈ 0.36 × 61% ≈ **22%**
- `block`（attn + FFN + norm 都在短序列上）：≈ 0.36 × 61% + 0.2 × 39% ≈ **30%**

实测 wall-clock 收益还要扣除 matching / gather / scatter 开销，**Phase 4/5 的验收阈值据此设定**（§7），不许拍脑袋。若实测远低于理论值，先用 profiler 定位 merge 开销而不是直接加大 ratio。

### 2.2 三个 merge 作用域（config 可选）

- `scope = "pre_attn_restore"` — 仅调试。merge → attn → restore，验证 matching/restore 数学。
- `scope = "attn_only"` — 中间里程碑。仅 attention 在短序列上计算，FFN 全长。用于隔离验证 RoPE/索引手术的正确性。
- `scope = "block"` — **主交付物**。在 block 入口（norm1 之后）merge 视频 token，attention、FFN、modulation 全部在短序列上运行，block 出口 unmerge 回全长（残差连接在全长序列上完成，即：unmerge 后再加残差，保证未参与 merge 的 block 看到完整 layout）。

Stretch goal（明确不承诺）：`scope = "span"` — 连续多个 block 维持 merged 状态，仅在 span 两端 merge/unmerge，可摊薄 matching 开销。仅在 Phase 5 提前完成时尝试。

### 2.3 RoPE 位置策略（v3：从"半句话"升级为显式设计点）

Merged token 是多个空间/时间位置的加权平均，3D RoPE 需要一个位置。这是本项目真正的 research question 之一，实现两种策略并消融：

- `rope_mode = "pre_rope"`（默认）— 先对全长 Q/K 施加 RoPE，再按 merge 方案对 rotated 后的 Q/K 做加权平均。不需要为 merged token 选位置，位置信息以"旋转后向量的凸组合"形式保留。代价：RoPE 计算仍是全长的（RoPE 本身很便宜，可接受）。
- `rope_mode = "dst"` — merged token 直接使用 dst token 的位置编码。最简单，spatiotemporal 模式下跨帧 merge 时失真最大，作为消融下界。

消融安排在 Phase 7（筛选集上 r=0.2 对比两种 mode 的 CLIP + temporal 指标），赢家进终选矩阵。报告中必须有一段讨论此设计点。

### 2.4 Proportional attention（v3 新增，必须实现）

吸收了 s 个 token 的 merged key 在 softmax 中应有 s 票。实现：对 attention logits 加 `log(s)` 偏置（ToMe 标准做法），经 SDPA 的 additive `attn_mask` 传入。

**已知代价**：带任意 bias 的 SDPA 无法走 flash 后端，会回落到 memory-efficient/math kernel，可能吃掉部分加速。处理方式：`prop_attn` 作为 config 开关，Phase 7 实测 on/off 的速度差与质量差，**两个数字都进报告**；若速度代价 > 质量收益，默认关闭并如实说明。这正是 honesty contract 第 5 条的用法。

### 2.5 Matching 细节（v3 补全 v2 的空白）

- `match_feature = "hidden_norm"`（默认）— 在 norm1 输出上计算 cosine 相似度。原版 ToMe 用 attention K，但 block-scope merge 发生在 attention 之前，存在鸡生蛋问题；用 normed hidden 是务实选择。`"attn_k"`（缓存上一层 K）列为可选消融，时间不够就砍。
- **CFG batch 处理（v2 完全没定义）**：cond/uncond 是同一 batch 的两个元素，内容不同，matching 结果必然不同。规定：**逐 batch 元素独立 matching**，所有索引张量带 batch 维 `[B, r]`，merge/unmerge 用 batched gather/scatter。单元测试必须覆盖 B≥2 且两个样本内容不同的情形。
- `protect_first_frame` 语义澄清：保护的是 **latent 帧 0**（对应像素域前若干帧）。报告中按此措辞，不许写成 "first video frame"。

## 3. Module Design

### 3.1 `src/tokmerge/merging.py`（纯张量，无 diffusers 依赖）

```python
from dataclasses import dataclass
import torch

@dataclass
class MergeConfig:
    enabled: bool = False
    ratio: float = 0.0
    mode: str = "spatial"               # "spatial" | "spatiotemporal"
    scope: str = "block"                # "block" | "attn_only" | "pre_attn_restore"
    rope_mode: str = "pre_rope"         # "pre_rope" | "dst"
    prop_attn: bool = True              # log(size) bias in softmax
    match_feature: str = "hidden_norm"  # "hidden_norm" | "attn_k"
    layers: tuple[int, ...] = ()
    temporal_window: int = 1
    protect_first_frame: bool = True    # latent frame 0

@dataclass
class RestoreInfo:
    src_idx: torch.Tensor   # [B, r]
    dst_idx: torch.Tensor   # [B, r]
    keep_idx: torch.Tensor  # [B, N-r]
    sizes: torch.Tensor     # [B, N-r]  merged token weights
    num_video_tokens: int
    grid: tuple[int, int, int]

def bipartite_soft_match(metric, r, frames, gh, gw, mode,
                         temporal_window, protect_first_frame):
    """metric: [B, N, C] L2-normalized. Per-batch-element independent
    matching. Returns batched (src_idx, dst_idx, keep_idx)."""

def merge_tokens(x, info) -> torch.Tensor:
    """Size-weighted average. [B, N, C] -> [B, N-r, C]."""

def unmerge_tokens(merged_x, info) -> torch.Tensor:
    """Scatter back to [B, N, C]; absorbed tokens copy dst value."""

def size_log_bias(info, num_text_tokens, num_heads) -> torch.Tensor:
    """Additive attention bias: log(sizes) on merged-video key positions,
    zeros on text keys. Shape broadcastable to SDPA attn_mask."""
```

硬性要求（v2 全保留，新增两条）：

- ratio=0 严格恒等（`allclose`）；任意 `B, frames, gh, gw, C`；确定性。
- **新增**：batched 索引，B 维内各样本独立正确。
- **新增**：`size_log_bias` 数值正确性单测（手算 2-token 例子）。

## 8. Experiment Matrix + 算力预算（v3 新增预算）

**Stage A 筛选**（Phase 7）：~10 config × 5 prompts × 1 seed = **50 runs**。
**Stage B 终选**（Phase 8）：baseline + {spatial, st} × {0.1, 0.2, 0.3} + 1 个 layers 对照 = 8 config × 10 prompts × 2 seeds = **160 runs**。

预算估算（quality preset，sequential offload 单次按 ~25 min 计，请用 Phase 0 实测值替换）：

| Stage | Runs | GPU-hours（估） |
|---|---|---|
| A 筛选 | 50 | ~21 h |
| B 终选 | 160 | ~67 h |
| VBench 评测 | — | ~10 h |
| **合计** | | **~98 h** |

**降级预案（日历不够时按顺序砍，写死在此处避免临场乱砍）**：

1. Stage B seeds 2→1（省 ~33 h）

≥24GB GPU 若可借到：仅复跑 baseline + 默认 r=0.2 两个 config（4 runs）拿干净 GPU speedup 数字，不复跑全矩阵。

## 13. v2 → v3 变更追溯表（给评审/队友看）

| v2 缺陷 | v3 对策 | 落点 |
|---|---|---|
| 只砍 attention，FFN 全长 | `scope="block"` 为主交付物 + FLOPs 预算 | §2.1, §2.2, Phase 5 |
